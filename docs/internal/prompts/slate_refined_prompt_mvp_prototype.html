SLATE — Refined product requirements prompt

Updated for: both modes equally · all NLEs · dual-mode data model · complete copy-ready prompt.

Key architectural decision added: Narrative and documentary are not two separate apps. They are two organizational models on top of one shared media engine. The data model uses a universal "Clip" object with a flexible metadata hierarchy that adapts to either mode.

Dual-mode data model (new)

Narrative mode hierarchy

Footage is organized by the script breakdown structure.

  • Project → Episode/Reel → Scene → Setup → Take
  • Slate metadata: scene, shot, take number, director, DP
  • Rough cut assembles by scene → shot coverage → best take per setup
  • Script upload enables flub detection and line coverage reporting
Documentary mode hierarchy

Footage is organized by who speaks and when.

  • Project → Shooting Day → Subject → Clip
  • Metadata: subject name, location, interview session, topic tags
  • Rough cut assembles by subject → topic → AI-scored best moments
  • Transcript search across all clips enables paper-cut workflow
Shared underneath: Both modes use the same Clip object (media path, proxy path, duration, timecode in/out, audio tracks, AI scores, annotations, status). Only the parent hierarchy and display logic differ. Users can switch a project between modes without re-ingesting.

Refined AI scoring spec (both modes)

Narrative-specific scoring

  • Script comparison: transcript vs uploaded script text
  • Line coverage: did subject complete the full line?
  • Performance energy: prosody, speech rate, pauses
  • Continuity flag: check eyeline, wardrobe, props vs prior take
  • Composite weights: visual 40% · audio 30% · performance 30%

Documentary-specific scoring

  • Content density: unique information per minute of speech
  • Subject clarity: face detection + eye contact + frame stability
  • Moment detection: emotional register peaks, emphasis detection
  • Duplicate detection: flags clips covering same ground as prior
  • Composite weights: visual 30% · audio 30% · content 40%

Phase 1 MVP — 8-week scope

Everything a DIT or post coordinator needs to get synced, organized dailies to a director review link. No AI scoring yet — that ships in Phase 2. Ship something real, fast.

Phase 1 philosophy: The bottleneck on set isn't AI — it's the hour it takes to manually sync audio, rename files, and build bins. Phase 1 eliminates that entirely. A single watch-folder ingest + auto-sync gets you to an organized, proxy-playable bin in under 10 minutes. AI scoring layers on top in Phase 2.

Phase 1 success criteria

Speed targets

  • Card mount → proxies ready: under 20 min for a 2-hour card
  • Audio sync: under 2 min per hour of footage
  • Rough assembly export: under 60 seconds for a full shoot day
  • Review link live: under 5 min after assembly

Quality targets

  • Sync accuracy: ≥98% correct on first attempt
  • Metadata parse accuracy: ≥95% on standard naming conventions
  • Zero data loss: checksums verified on 100% of ingested files
  • Web portal playback: ≤2s load on standard broadband

Product gaps & solutions

Six gaps identified vs. production-level requirements. P1 = Phase 1 addition (must fix before beta). P2 = Phase 2 addition (data model must accommodate now).

The data model is the constraint. P2 features can wait to be built, but the schema must accommodate them now. Schema migrations after beta distribution are painful. Add the fields now, implement later.
1
Multi-camera sync P1 — Phase 1 addition
Almost every TV drama shoots 2–4 cameras simultaneously. Single-camera sync forces a DIT to run the workflow N times.
The current syncClip(videoURL, audioFiles, fps) takes one video. Multi-cam requires a syncCameraGroup(cameras[], audioFiles[], fps) that syncs all angles to a shared timecode reference in a single pass and stores per-angle offsets.
Schema addition (NarrativeMeta)
"cameraGroup": { "groupId": "string", "angle": "A"|"B"|"C"|"D", "cameraBody": "string", "lens": "string", "isMainAngle": boolean }
"cameraGroupSyncResult": { "groupId": "string", "offsets": [{ "angle": "A", "offsetFrames": 0, "driftPPM": 2.1 }] }
2
Notification & delivery hook P1 — Phase 1 addition
The 20-minute promise ends with a DIT manually copy-pasting a URL. The last mile must be automated.
On assembly completion, SLATE auto-generates the review link and sends it to the director via iMessage (or email fallback). DIT configures delivery targets per-project. Status shows in the desktop status bar: "Review link sent to Sarah (Dir) 2m ago."

Architecture fixes

Five technical gaps — two are pre-beta blockers. Must be resolved before any real-production shoot day test.

Blockers 1 and 2 (IPC + crash recovery) block beta. Without XPC, the desktop app cannot observe daemon progress. Without crash recovery, a sleep/crash during ingest silently corrupts the project. Do not schedule a beta shoot day until both are green.
1
IPC: NotificationCenter → XPC BLOCKER
NotificationCenter is in-process only. The daemon and app are two separate OS processes. The notification never arrives.
XPC (Apple's standard IPC for macOS) is the correct solution. The daemon exposes a named XPC service (com.mountaintoppics.slate.ingestd). The app connects via NSXPCConnection and receives IngestProgressReport objects over the channel.
XPC Protocol (new file: IngestXPCProtocol.swift)
@objc protocol IngestXPCProtocol {
  func progressDidUpdate(_ report: IngestProgressReportXPC, withReply: @escaping () -> Void)
  func registerWatchFolder(_ config: WatchFolderConfigXPC, withReply: @escaping (Bool) -> Void)
  func pauseIngest(withReply: @escaping () -> Void)
}
// App-side: IPCManager.swift connects via NSXPCConnection, @Published var progressReport
2
Ingest crash recovery state machine BLOCKER
Ingest state lives in memory. A crash or sleep leaves orphaned Clip records. No resume on next launch.
All ingest state must be persisted to GRDB before each stage begins. On next launch, the daemon reads any records stuck in non-terminal stages (copying, proxy_active) and resumes them automatically.
GRDB: ingest_queue table
CREATE TABLE ingest_queue (
  id TEXT PRIMARY KEY, clip_id TEXT, source_path TEXT, destination_path TEXT,
  stage TEXT DEFAULT 'queued',   -- queued|copying|checksumming|proxy_pending|proxy_active|sync_pending|ready|failed
  stage_started_at REAL, attempts INTEGER DEFAULT 0, last_error TEXT,
  created_at REAL, updated_at REAL
);
-- On launch: records with stage='copying' older than 5min → reset to 'queued'. Retry automatically.
3
Commit to Cloudflare R2 as cloud storage backend
R2 is already assumed in 3 edge functions. The "open question" is already decided by the code.
Formalize: add contracts/storage.md documenting bucket convention ({projectId}/{clipId}/proxy.mp4), presigned URL expiry (24h review / 7d download), upload path from daemon. R2 egress is $0 vs AWS S3's $0.09/GB — correct choice given proxy streaming volume.
4
Desktop CI pipeline
Web has CI. Swift packages have test scripts. The desktop app has no CI — a broken @main conflict won't be caught.
.github/workflows/desktop-ci.yml (key steps)
runs-on: macos-14  # Apple Silicon runner
xcodebuild build -scheme SLATEDesktop -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO
bash scripts/test-sync.sh
python3 -m json.tool contracts/*.json  # validate all contracts are valid JSON
5
Code signing & notarization — required before beta
macOS Sequoia blocks unsigned apps on first launch. Beta users on clean Macs can't open SLATE without this.
Steps: Developer ID Application certificate → store as GitHub secret → add xcrun notarytool submit to CI after archive → configure Hardened Runtime entitlements (watch folders, network, iMessage scripting). ~3 min per build. Must be done before distributing any .dmg.

Assembly logic — both modes

Narrative assembly
  • Iterate scenes in scene-number order
  • For each setup: use highest-scoring circled take
  • No circle → use highest AI score, flag amber "auto-selected"
  • No takes → mark red "missing coverage", continue
  • Assembly = ordered array of Clip references + gaps
Documentary assembly
  • Group clips by subject
  • Within each subject, group by topic tag
  • Circled clips → include all; flagged → include, mark review
  • Sort within topic by AI content score
  • B-roll appended after each subject block
Auto-share on export: When the export button is pressed, SLATE immediately calls generate-share-link and fires NotificationService.deliver() without requiring the DIT to do anything extra. The "sent" status appears in the status bar within 5 seconds.
